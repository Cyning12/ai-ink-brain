# 从 RAG 到 Agentic Search，再到“可验证的架构认知”：我的 AI 协作落地实践

## 前言

我在全栈项目里长期使用 AI 辅助开发，经历过三个阶段：

- **阶段 A**：靠零散注释、口口相传的“经验文档”协作
- **阶段 B**：全面投入 RAG（把知识切片进库，再检索回答）
- **阶段 C**：转向 **Agentic Search（动态探索）+ 架构认知的静态收敛（可验证）**

很多人会把「代码库协作」默认等同于 RAG，但我在长期对比实践后更认可这个结论：**代码工程场景里，Agentic Search 更贴近开发者直觉；而要把它做成可持续协作，还必须补上一层“静态且可验证的架构事实源”。**

这篇文章只谈概念与方法，不引用任何项目内部文件名、目录名或脚本路径；目标是给你一套可以迁移到任意工程的认知框架与落地清单。

---

## 1) RAG 与 Agentic Search：本质差异是什么？

### 1.1 传统 RAG 的工作模式（更像“资料问答”）

经典 RAG 的范式是 **前置加工 + 单次检索 + 生成**：

- **预处理**：切片、Embedding、入库
- **检索**：按 query 召回 Top-K 片段
- **生成**：拼上下文让模型回答

它在“静态资料库”上很强：手册、规章、产品文档、历史资料，都属于这类。

但在“高频迭代的代码工程”里，RAG 会遇到一些结构性阻力：

- **索引滞后**：代码变了、索引没变，回答自然容易读到“昨天的真相”
- **语义碎片化**：切片把调用链、边界与意图切断，模型看到的是碎片而不是系统
- **符号精确性**：代码世界的关键是“名字与引用”（函数名、路由、变量、表/字段），模糊语义匹配反而可能引入无关片段
- **维护成本**：入库/更新/清理/版本一致性，是额外工程系统

### 1.2 Agentic Search 的工作模式（更像“人类查代码”）

Agentic Search 是 **模型自主规划 + 工具实时检索 + 多轮迭代校正** 的主动探索范式。它更接近人类开发者的真实行为：

- 先圈范围（目录、模块、入口）
- 再定位符号（路由、函数、常量、表名）
- 然后追链路（调用关系、错误分支、依赖）
- 信息不够就继续查，直到足够再输出结论或改动

在代码场景里，它的关键优势不是“更聪明”，而是**更符合事实获取方式**：读当前代码、追当前依赖、对齐当前状态。

### 1.3 一句话总结差异

- **RAG**：先“存档”，再“问答”
- **Agentic Search**：先“探索”，再“归纳”

## 2) Agentic Search 的优势与短板：为什么它需要“静态收敛层”？

### 2.1 优势（代码工程里很关键）

- **实时一致**：不依赖索引同步，改完代码立刻可被检索到
- **上下文更精确**：模型会选择“缺什么补什么”，而不是固定塞 Top-K 碎片
- **结构感更强**：目录与命名本身就是架构信号，能帮助快速建立模块边界
- **链路更短**：少一层向量中间件，排错与追因更直接

### 2.2 短板（也是协作灾难的根源）

Agentic Search 只负责“探索”，天然会碎片化，长期协作会出现三类问题：

- **全局共识缺失**：每次都从零探索，团队对“系统怎么跑”缺少稳定统一口径
- **边界容易漂移**：今天为了快改一处，明天又改另一处，长期架构会被侵蚀
- **排障成本高**：如果你只描述业务 Happy Path，出错时“最短定位链路”不存在，新人/新 Agent 就只能猜

结论是：**Agentic Search 适合解决“局部事实获取”，但不擅长提供“全局、长期、可验证”的架构共识。**

---

## 3) 关键融合：动态探索 + 静态事实源（可验证）

我把这层静态收敛称为 **Tech Graph（技术图谱）**，你也可以理解为：

- Architecture Graph（架构图谱）
- System Graph（系统图谱）
- Runtime Graph（运行图谱）

它不是某个目录名，更不是“画图做样子”。它的作用只有一个：**把团队需要长期一致的“系统事实”，沉淀成可以被检查、可以被追踪、可以被新人复用的结构化认知。**

### 3.1 两层分工（非常清晰）

- **动态探索层（Agentic Search）**：解决“现在要改哪里、具体怎么改、实现细节是什么”
- **静态收敛层（Tech Graph）**：解决“系统边界是什么、链路怎么跑、失败时怎么定位、哪些不允许越界”

### 3.2 静态收敛层应该收敛什么？

如果你只记住一个清单，我建议是这四类：

- **入口与分支**：系统有哪些入口？有哪些关键分支（例如：业务链路 A/B/C）？
- **数据结构与契约**：核心数据模型、接口契约、表/字段的最小真值
- **失败路径**：鉴权失败、依赖不可用、0 命中、超时、降级策略……必须显式画出来
- **运行观测**：关键事件/日志落点、错误分层、超时与重试、可观测字段（仅字段名即可）

### 3.3 最容易踩坑的误区：把它写成“纯文档”

静态收敛层一旦变成“写给人看的长文档”，很快就会：

- 跟代码脱节（静默过期）
- 变成经验主义（解释很多但缺乏约束）
- 让新人读完仍然找不到定位路径

更好的方式是：**结构化 + 可验证**。你可以不做复杂的工具链，但至少做到：

- 能自动检查“关键事实是否与代码一致”（例如：入口、对外接口、关键依赖名）
- 能让人用最短路径完成排障（例如：看到错误阶段就知道下一步看哪里）

## 4) 我如何判断“这套体系落地了”？

我用的是非常工程化的验收口径，而不是“图画得很漂亮”：

- **新人/新 Agent 接手**：只读“入口总览 + 规约 + 失败路径 + 运行观测”，就能定位到正确的入口与边界
- **改动可追踪**：当你改了关键事实（入口、接口、数据模型、关键依赖名），静态事实源必须同步，否则会被拦截
- **排障可缩短**：遇到问题时，不依赖“猜”，而是按失败路径图走最短定位链路

---

## 5) RAG 还有用吗？我的场景化答案

**有用，但别拿它做“代码协作的默认答案”。**

- **代码工程**：优先 Agentic Search，配静态事实源收敛共识与边界
- **海量静态资料**：RAG 仍然非常强，尤其是长文档/历史资料/产品手册

你可以把它理解为：**RAG 适合“资料检索”，Agentic Search 适合“工程探索”。** 两者不是敌人，只是适用域不同。

---

## 结尾：一句话的落地建议

如果你正在把 AI 带进工程协作，我建议从这条路径开始：

- 先用 **Agentic Search** 把“事实获取”做对（实时读代码、追链路）
- 再用 **Tech Graph** 把“长期共识”收敛（入口/契约/失败路径/运行观测）
- 最后把静态事实源做成 **可验证**（至少能发现关键事实漂移）

这不是更复杂的工程，而是更少幻觉、更可持续的协作方式。
